Docker & Linux
on macOS, unboxed.

A lightweight, OrbStack-style runtime that boots a Linux VM with Apple's Virtualization.framework — then projects the Docker engine, file sharing, x86 translation and port mapping straight onto macOS. No Docker Desktop.

Named for the Oort cloud — the vast shell of countless small icy bodies at the edge of our solar system. Here: the many lightweight machines & sandboxes orbiting one shared kernel.

What works today

A working clone — verified on real hardware.

oort studies why OrbStack is fast and light, then reimplements the core mechanisms by hand. One command to start; the stock docker CLI just works.

🐳

Docker over vsock ✓

VZ boots a Linux VM; dockerd is projected onto a macOS Unix socket through virtio-vsock.

🌐

Networking ✓

dockerd manages iptables NAT — docker build and runtime egress work; guest follows the Mac's DNS.

📁

VirtioFS sharing ✓

Bind-mount Mac directories straight into containers over a VirtioFS share.

⚡

Rosetta x86-64 ✓

--platform linux/amd64 images run via Apple's Rosetta translation.

🔌

Port mapping ✓

Published container ports surface on localhost automatically.

🪶

zram + ballooning ✓

Kernel-tier memory efficiency: zram swap plus active balloon that returns idle RAM to macOS.

☸️

Kubernetes (k3s) ✓

One command runs k3s in the guest and projects the kube API onto localhost:6443.

📦

Linux machines ✓

Named, multi-distro environments you shell into — Ubuntu, Alpine, more.

🖥️

Menu-bar GUI ✓

A native SwiftUI menu-bar app shows status and starts/stops the engine.

Quick start

From zero to docker run in four steps.

Requires macOS 13+ on Apple Silicon. The CLI is oort (so it never clashes with OrbStack's own orb).

1

Clone & build the golden image

One-time: build and provision the disk, then snapshot a ready-to-boot "golden" image.

zsh — ~/devcopy
$ git clone https://github.com/everettjf/oort
$ cd oort
$ ./oort build-image       # provisions once, snapshots golden
done. 'oort start' is now instant; 'oort reset' restores this snapshot.
2

Start the VM

Boots Docker, file sharing, Rosetta and port forwarding. A reused golden disk is ready in seconds.

zsh — ~/dev/oortcopy
$ ./oort start
starting oort VM…
waiting for Docker.... ready.
$ eval "$(./oort env)"   # export DOCKER_HOST=unix://~/.oort/docker.sock
3

Use the stock Docker CLI

It's just Docker. Run containers, publish ports, even run x86-64 images via Rosetta.

zsh — dockercopy
$ docker run --rm hello-world
Hello from Docker!
$ docker run -p 8080:80 nginx        # curl localhost:8080 on the Mac
$ docker run --platform linux/amd64 alpine uname -m
x86_64                                # via Rosetta
4

Go further — machines & Kubernetes

Spin up named Linux machines, or bring up a k3s cluster with a single command.

zsh — oortcopy
$ oort machine create dev ubuntu   # named Linux machine
$ oort machine shell dev
$ oort k8s enable                  # k3s in the guest
$ kubectl get nodes
NAME      STATUS   ROLES                  AGE   VERSION
oort   Ready    control-plane,master   12s   v1.30…
How it works

Two halves: a Swift engine and a Go guest agent.

VZ

Swift virtualization engine

Builds a VZVirtualMachineConfiguration — virtio-blk root disk, NAT net, vsock, VirtioFS, Rosetta share, memory balloon — and owns the lifecycle on a dedicated VM queue.

↔

vsock socket projection

Each host Unix socket is a tunnel to a guest vsock port: dockerd on 2375, the exec agent on 2376, TCP forwards on 2377.

Go

Static guest agent

A tiny static linux/arm64 binary splices the Docker socket, runs oort exec commands, and forwards published ports — steady under load where Python wasn't.

🔒

Durability by design

An exclusive flock on the disk image stops two VMs corrupting it; graceful-only shutdown; oort reset restores the golden snapshot instantly.

◇ Honest status

A cold boot from the golden image is fast and reliable (~8.8 s), and Docker, VirtioFS, Rosetta, machines, zram, ballooning and DNS-follow are verified end-to-end. oort is a study project, not a Docker Desktop replacement for production.

Known residual: restarting a heavily-mutated disk can occasionally leave the guest services down — oort reset restores the golden image as a reliable recovery. The remaining moat (OrbStack-class cold start and native-speed bind mounts) is the deliberately-deferred hard work. See the plan for the full roadmap.
