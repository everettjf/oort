# 计划：追平 OrbStack

[English](./plan.md) | [**简体中文**](./plan.zh-CN.md)

一份一步步追平 OrbStack **速度与体验**的可执行计划。每个里程碑都能独立交付、有可量化的追平目标、并说明如何验证。大致按顺序推进：先把速度/效率的基础打好，再补体验，最后啃文件系统这块最硬的护城河。

> 基线数字来自 `bench.sh`。每个里程碑做完后重跑它来追踪进度。

## 重新评估 —— 与 OrbStack 的真实差距

做完 M1–M8 后,判断变了:最大的差距**不再是某个缺失功能**,而是**地基还不够稳 + 一批功能写了但没被证明**:

- **可靠性**:这一轮反复遇到 VM 引导/首次配置超时、强杀弄脏盘、残留构建进程占着 SwiftPM/Go 锁导致连环卡死。OrbStack 开箱即用,而 openorb 有时还"起不来"——这是最痛的实际差距。
- **未验证的工作**:`orb route`(M5)、k3s(M6)、机器(M7)代码完整但从没端到端跑过——而且 M7 还真藏了个 bug,直到一个迟到的验证任务跑完才暴露。**没跑过的代码不算数。**
- **成熟度**:GUI(M8)只是菜单栏壳;没有打包分发;文件系统护城河(M9)没碰。

差距归为三类:

**A. 地基(稳定性 & 可信度)—— 当前最大短板**
1. 引导/配置可靠性:`orb start` 必须*总能*起来,关机绝不脏盘,不出现锁连环卡死。
2. 把写好的验通:M5 route、M6 k3s、M7 机器跑出真实端到端绿灯(并修 bug)。
3. 自动化 e2e 测试套件(start→docker→挂载→rosetta→端口→k8s→机器→stop),接 CI,杜绝"未验证的提交"。

**B. 性能(真"快")**
4. 自编译精简内核 + 直引导 → 冷启动从 ~8s 压到 ~1–2s(OrbStack 水平)。
5. **M9 文件系统护城河** —— 自研 virtiofs 缓存/DAX(很可能要在 `Hypervisor.framework` 上自建 VMM)。小文件从 ~21× 干到接近原生。唯一研究级难点。

**C. 产品(真"好用")**
6. 完整 GUI(容器/镜像/卷/日志/机器/k8s/设置),而非菜单栏壳。
7. 网络纵深:VPN *流量*路由(gvproxy)、容器按 IP 直达、`*.orb.local` 域名。
8. 打包:签名+公证 `.app`、dmg / brew cask、Sparkle 自动更新。

## 推进方案 —— 阶段 0 → 1 → 2

> 相比原来的 M 顺序做了**调整:先夯地基,再冲性能,最后产品**。地基没稳就继续堆功能 = 沙上建塔。

**阶段 0 —— 地基硬化(先做)**
- 0.1 根因排查并修复引导/配置可靠性(EFI 延迟?cloud-init 卡 apt?锁竞争?);目标稳定 <10s 起;关机一律优雅路径;`orb start` 自检自愈。
- 0.2 `make verify` 一键端到端套件 + CI;任一失败即红。
- 0.3 验通并加固 M5 route / M6 k3s / M7 机器,做到真绿。

**阶段 1 —— 性能**
- 1.1 自编译精简内核 + 直引导 → ~1–2s 启动。
- 1.2(护城河)M9 自研 virtiofs 缓存/DAX → 文件系统接近原生。最难最贵,压轴。

**阶段 2 —— 产品**
- 2.1 完整 SwiftUI 应用(多面板)。
- 2.2 gvproxy 网络栈(VPN 流量 + 容器 IP + 域名)。
- 2.3 签名/公证 `.app` + brew cask + 自动更新。

**结论:** 打平 OrbStack 的*速度* ≈ 阶段 1.2(研究级、数月、很可能要自建 VMM);打平*稳定与精致* ≈ 阶段 0 + 阶段 2(工程量大但不难)。**下一步该做阶段 0,而不是 M9。**

## M1 — 秒级重启（golden 镜像）
- **目标：** 在已配置好的盘上 `orb start` **≤ 3 秒**;不再重复 provisioning。
- **任务：** 让 `orb build-image` 产出完整配置好的 "golden" 盘(启动一次、等 cloud-init 跑完、快照);`orb start` 直接复用;保留一份纯净副本以便低成本重置。
- **验证：** `time orb start` ≤ 3 秒;紧接着 `docker run hello-world` 立即可用。
- **工作量：** 🟢 小 · **风险：** 低 · **依赖：** 无。

## M2 — 快速启动 + 自编译内核
- **目标：** 冷启动向 OrbStack 的 ~1–2 秒靠拢;有一个我们可控的内核。
- **任务：** **在客户机内**编译内核(原生 arm64,免去 macOS 交叉编译的坑),单体内核(virtio-blk/net/fs、vsock、ext4、overlay、**zram** 全设 `=y`);用 VZ **直引导**(`--kernel`,已支持)启动而非 EFI;裁掉无用驱动。
- **验证：** `uname -r` 显示自定义内核;`bench.sh` 测启动时间;完整 Docker/VirtioFS/Rosetta/端口转发套件仍通过。
- **工作量：** 🟡 中 · **风险：** 中(配出能启动的 `.config` 可能要试几次) · **依赖：** 无。
- **解锁：** zram 内置(M3)、调优、更快启动。

## M3 — 内存效率（zram + ballooning）
- **目标：** 空闲占用低;像 OrbStack 一样按需回收内存。
- **任务：** 启用 **zram** 交换(M2 内置);在主机侧加一个**主动 ballooning** 循环,读客户机内存压力并调整 VZ 气球目标;可选 **KSM** 做跨容器内存页去重。
- **验证：** 空闲 RSS 明显下降;跑很多小容器不吃满内存。
- **工作量：** 🟡 中 · **风险：** 中 · **依赖：** M2(zram)。

## M4 — 开发文件系统（务实版）
- **目标：** 现在就让"挂源码开发"可用(不等 M9 大改)。
- **任务：** 给 `orb` 加助手 + 文档,推命名卷工作流(把 `node_modules`/构建目录放客户机本地盘);调研客户机侧任何可用的缓存开关;**给 `bench.sh` 加小文件 + 真实负载(`npm install`)用例**。
- **验证：** 文档化的工作流让 `npm install` 接近原生;bench 显示提升。
- **工作量：** 🟢 小–中 · **风险：** 低 · **依赖：** 无。

## M5 — 无缝网络（用户态网络栈）
- **目标：** VPN **流量**路由、容器/机器按 IP 互达、可选域名。
- **任务：** 集成用户态网络栈(如 **gvisor-tap-vsock / gvproxy**),让客户机流量经 macOS 走(跟随 VPN 路由),配统一 bridge。
- **验证：** 容器能访问只在 VPN 内的主机;macOS 上能 ping 通容器 IP。
- **工作量：** 🔴 大 · **风险：** 中 · **依赖：** 无(DNS 解析已跟随 Mac)。

## M6 — Kubernetes
- **目标：** 一条命令拉起可用集群。
- **任务：** 在客户机跑 **k3s**;投影 kube API(vsock tcp 转发 → `localhost:6443`);写入/合并 kubeconfig;`orb k8s enable/disable`。
- **验证：** macOS 上 `kubectl get nodes` 并部署一个示例。
- **工作量：** 🟡 中 · **风险：** 低–中 · **依赖：** M5 锦上添花(服务网络)。

## M7 — 多台 Linux 机器
- **目标：** OrbStack 式的命名、多发行版机器,共用内核。
- **任务：** 管理多个 rootfs 镜像;`orb machine create/list/delete/exec`;每机器独立挂载/网络;一个 SSH/exec 多路复用。
- **验证：** 创建 Ubuntu 和 Alpine 两台机器,分别 exec 进去,文件隔离。
- **工作量：** 🔴 大 · **风险：** 中 · **依赖：** M2(共用自定义内核更顺)。

## M8 — GUI
- **目标：** 原生应用体验,而非纯 CLI。
- **任务：** 一个包裹引擎的 **SwiftUI** 应用——状态/菜单栏、容器、机器、日志、设置、启停;打包 `openorb` 引擎 + 客户机 agent。
- **验证：** 安装 `.app`,从界面管理一切。
- **工作量：** 🔴 大 · **风险：** 低(是工程量,不是研究难题) · **依赖：** M6/M7 让功能面更完整。

## M9 — 护城河：原生级文件系统
- **目标：** 追平 OrbStack 的 ~75–95% 原生 bind mount 速度(现在慢约 21×)。
- **任务：** 最硬的部分——VZ 内置 VirtioFS 不暴露缓存/DAX 调优,所以需要我们**自己的 virtiofs 服务端,带缓存/批处理(可能含 DAX)**。这很可能意味着放弃 VZ 高层 API,改用 **`Hypervisor.framework` 上的自建 VMM**(以掌控设备模型)。这是最深、最贵的一块——OrbStack 真正的护城河。
- **验证：** `bench.sh` 小文件落在本地盘的 ~2× 以内;bind mount 上 `npm install` 接近原生。
- **工作量：** 🔴🔴 非常大(研究级难度,数月) · **风险：** 高 · **依赖：** 最好在 M1–M8 之后。

## 里程碑 vs 阶段

上面的 M1–M9 是最初的功能拆解(M1–M8 已建好,见「进度」)。而**阶段 0 → 1 → 2** 才是经过重新评估后**接下来真正执行的计划**:阶段 0 把"建好了但不稳"的地基变可信,阶段 1(含 M9)冲真速度,阶段 2 补产品面。OrbStack 深入分析见[调研报告](../orbstack-research.md)。
